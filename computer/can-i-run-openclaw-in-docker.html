Can i run openclaw in docker

27 minutes ago

Yes, you can run

OpenClaw in Docker. In fact, containerization is the recommended and most stable way to deploy it because it isolates the agent's autonomous actions from your host operating system.

How to Run OpenClaw in Docker

The official repository includes a setup script that automates the entire process, including building the image and running an onboarding wizard.

  1. Clone the Repository: 
    git clone https://github.com/openclaw/openclaw.git
cd openclaw
  2. Run the Setup Script: 
    ./scripts/docker/setup.sh
    This script builds the gateway image, generates a gateway token, and starts the services via Docker Compose.
  3. Access the Dashboard:
    Open http://127.0.0.1:18789/ in your browser and use the token generated during the setup to log in.

Key Considerations

  • System Requirements: Your machine (or VPS) should have at least 2 GB of RAM for the initial build process, as pnpm install can be memory-intensive.
  • Security: Running in Docker provides a "safety net," ensuring the agent only interacts with files and services you explicitly allow. For enhanced security, you can use Docker Sandboxes to isolate tool executions in micro-VMs.
  • Persistence: By default, OpenClaw uses bind-mounts to ensure your configurations and workspace data survive container restarts.
  • Local Models: If you want to avoid cloud API costs, you can connect OpenClaw to a local Ollama instance by setting the hostname to host.docker.internal (on macOS/Windows) or the bridge IP (on Linux).

To connect OpenClaw (running in Docker) to a local

Ollama instance, the most critical step is ensuring the container can reach your host machine's network.

1. Configure Ollama for Network Access

By default, Ollama only listens to 127.0.0.1. You must change this to 0.0.0.0 so the Docker container can "see" it.

  • Linux: Edit your systemd service or run export OLLAMA_HOST=0.0.0.0 before starting ollama serve.
  • macOS/Windows: In the Ollama app settings, ensure it is set to allow connections from your local network.

2. Set the Correct Base URL

When OpenClaw asks for the provider URL during the onboard process or in your .env file, do not use localhost. Use the Docker-specific host address:

  • Base URL: http://host.docker.internal:11434.
  • API Key: Use any placeholder value (e.g., ollama-local), as Ollama does not require a real key for local use.

Update your openclaw.json or .env file with these settings for the best experience:

  • LLM Provider: ollama.
  • Context Length: OpenClaw agents perform better with larger windows. It is recommended to set a context window of at least 64k tokens.
  • Recommended Models: Use models that support tool-calling natively, such as llama3.3, qwen2.5-coder:32b, or gpt-oss:20b.

4. Quick Start Command

If you have Ollama installed, you can use the official launcher which automates much of this setup:

ollama launch openclaw

This command detects if OpenClaw is installed and guides you through picking a local model and configuring the gateway.

If I use docker on a host can a virtual guest (virt-manager) access it, and can it use my gpu?

18 minutes ago

Yes, a virtual guest in

virt-manager can access Docker containers running on your host, but sharing a single GPU between them simultaneously is complex and often restricted by your hardware.

1. Accessing Docker from a Guest VM

A virtual guest can communicate with Docker containers on the host over the network.

  • Networking: If your guest and Docker use the same network bridge (e.g., virbr0), they can communicate directly via IP.
  • Exposing Services: To access a containerized service (like an API or dashboard), you must map the container's internal ports to the host's ports (e.g., -p 8080:80). The guest can then reach the service at http://<host-ip>:8080.
  • Shared Files: You can also share files between the host and guest using virtiofs or Samba, allowing the guest to see data used by Docker containers.

2. Using the GPU

Using your GPU in a virtual environment depends on how you allocate it:

  • GPU Passthrough (Full Control): You can assign a physical GPU directly to the guest VM using Virt-manager GPU Passthrough.
    • Limitation: Once passed through, the host (and thus any Docker containers on the host) typically loses access to that GPU.
  • Simultaneous Use (Sharing):
    • Consumer GPUs: Most standard NVIDIA/AMD cards do not support "splitting" between a host and a guest at the same time. You generally have to shut down the VM to release the GPU back to the host's Docker containers.
      • Enterprise GPUs: High-end cards (e.g., NVIDIA Tesla/Quadro) support vGPU or SR-IOV, which allows multiple VMs and the host to share one physical GPU simultaneously, though this often requires paid licenses.
  • Workaround: You can run Docker inside the guest VM. If you pass the GPU to the VM, every container running within that VM can use the GPU.

networking your guest to your host's Docker bridge

12 minutes ago

To network a

virt-manager guest directly to a Docker bridge, you have two primary methods: moving Docker onto the existing VM bridge or manually adding the Docker bridge to the VM's configuration.

Instead of connecting the VM to docker0, you can tell Docker to use the existing virbr0 bridge created by libvirt. This puts both your containers and your VM on the same subnet (e.g., 192.168.122.x).

  1. Configure Docker Daemon: Create or edit /etc/docker/daemon.json on your host: 
    {
  "bridge": "virbr0",
  "iptables": false
}
  2. Restart Services: 
    sudo systemctl stop docker
sudo systemctl restart libvirtd
sudo systemctl start docker
    Note: Existing containers may need to be recreated to pick up the new IP range.

Method 2: Manually Connect VM to docker0

If you want to keep Docker on its own network but let the VM "plug in" to it, you can add the docker0 bridge as a network interface in virt-manager.

  1. Open Virt-Manager: Select your VM and go to Open > Show virtual hardware details (lightbulb icon).
  2. Add Hardware: Click Add Hardware > Network.
  3. Select Bridge:
    • Network source: Choose Bridge device.
      • Device name: Manually type docker0.
  4. Fix Iptables (Crucial): Docker often blocks traffic from other bridges. Run these commands on the host to allow the VM to talk to containers: 
    sudo iptables -I DOCKER-USER -i virbr0 -j ACCEPT
sudo iptables -I DOCKER-USER -o virbr0 -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

Method 3: Shared Macvlan (Advanced)

You can create a macvlan network in Docker that uses the same physical interface as your VM's bridge. This gives containers their own IP addresses on your actual local network, making them appear as "real" devices to your guest VM.
