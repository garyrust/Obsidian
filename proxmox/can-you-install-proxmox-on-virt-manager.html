Yes, you can install Proxmox VE as a VM under KVM/QEMU using virt‑manager, and it works well for labs, learning, and light nested workloads if your CPU supports nested virtualization and you configure it correctly. For production or heavy nested workloads, running Proxmox directly on bare metal is strongly preferred due to performance and complexity trade‑offs.saturnme+2​

How it works in virt‑manager

You install Proxmox like any other OS in KVM: create a new VM in virt‑manager, attach the Proxmox ISO, assign CPU/RAM/disk, and complete the graphical installer inside the VM. For reasonable performance of VMs inside Proxmox, the KVM host must expose hardware virtualization extensions (VT‑x/AMD‑V) to the Proxmox VM (nested virtualization) and usually use a “host‑passthrough” CPU type.getlabsdone+3​

Key setup points:

  • Enable nested virtualization on the host (kernel module options for Intel/AMD).pve.proxmox+1​
  • In virt‑manager, set CPU model to “host-passthrough” (or similar) so Proxmox sees VT‑x/AMD‑V.saturnme+1​
  • Allocate enough resources: at least 2–4 cores and 4–8 GB RAM for a comfortable lab, plus disk space for Proxmox and its guests.proxmox+1​
  • Use bridged or VLAN‑aware networking if you want the Proxmox guests to appear on your LAN.ecintelligence+1​

Pros of Proxmox inside virt‑manager

  • Safe lab and learning environment
    • Great for experimenting with Proxmox clustering, storage, backups, HA concepts, and API without touching bare‑metal installs.pve.proxmox+1​
    • Easy to snapshot/clone the entire Proxmox node from the KVM host for rollback and testing.getlabsdone+1​
  • Convenient for laptop/desktop setups
    • Lets you run Proxmox on a single Linux workstation that already uses KVM/virt‑manager, without dedicating a whole box.servermania+1​
    • Good for short‑term PoCs, training, and trying different Proxmox versions or configurations side by side.proxmox+1​
  • Networking and tooling flexibility
    • You can build complex lab topologies using libvirt networks, bridges, and things like Open vSwitch, then have Proxmox sit inside that virtual lab.ecintelligence+1​
    • Integrates with existing KVM tooling/scripts if you already manage everything with libvirt and virt‑manager.servermania+1​

Cons and limitations

  • Performance overhead and nested virtualization issues
    • Every VM or container you run inside Proxmox becomes “nested” under KVM; without proper nested hardware virtualization, KVM guests inside Proxmox can be an order of magnitude slower and nearly unusable.pve.proxmox​
    • Even with nested extensions enabled, performance is still worse than running Proxmox directly on bare metal, especially for heavy IO or CPU‑bound workloads.servermania+1​
  • Not ideal for production
    • Proxmox’s own docs position nested virtualization for testing and learning, not for production clusters.proxmox+1​
    • Features like live migration and some advanced Windows VBS/nested scenarios can behave poorly or require careful tuning when nested.reddit+1​
  • Resource and complexity overhead
    • You incur two layers of scheduling and memory management: KVM host and Proxmox guest, which increases overhead and tuning complexity.pve.proxmox+1​
    • Storage stacks can become complex (e.g., LVM/ZFS on the host plus ZFS/Ceph/LVM in Proxmox), and mis‑configuring caching or layers can hurt performance or reliability.proxmox+1​

When it makes sense vs bare metal

  • Good use cases: homelab, certification prep, testing Proxmox features, building small virtual labs on a laptop/desktop, and trying configurations before applying them to real servers.pve.proxmox+1​
  • Use bare metal instead: any serious production environment, performance‑sensitive workloads, or scenarios where you want to maximize Proxmox features like clustering, live migration, and storage performance with minimal overhead.servermania+2​

Can Proxmox be installed on a USB NVME drive?

Yes, Proxmox VE can be installed onto an NVMe drive that is attached via a USB enclosure, as long as the system firmware (BIOS/UEFI) can see that USB device as a bootable disk and the Proxmox installer detects it.proxmox+2​

Basic feasibility

Proxmox is just Debian with a custom installer, so from its perspective a USB‑attached NVMe is just another block device. If, during installation, the target disk list shows your USB NVMe (often as /dev/sdX or /dev/nvmeXn1), you can select it and proceed; if your BIOS/UEFI can boot from that device, Proxmox will run directly from it.thomas-krenn+3​

Pros of USB NVMe for Proxmox

  • You get NVMe‑class performance with the flexibility of an external drive, which can be handy for small home labs or reusing hardware without opening the chassis.learnlinux+1​
  • It allows installing Proxmox on systems whose internal disks you do not want to wipe (for example, preserving an existing OS on the internal drive).proxmox+1​
  • USB NVMe enclosures often have much better endurance and speed than cheap USB flash sticks, making them more suitable for an OS that logs heavily.proxmox​

Cons and caveats

  • Boot support is the main limitation: many older boards either cannot boot from USB‑attached NVMe or are unreliable doing so; in those cases, a common pattern is bootloader on a small internal disk or SD card and Proxmox root/storage on NVMe.gist.github+1​
  • USB links add another point of failure: loose cables, power issues, or enclosure glitches can take the entire hypervisor down, which is risky for anything beyond a lab.proxmox+1​
  • Proxmox writes a lot of logs and metadata; while NVMe is fine with that, running entirely over USB still adds latency and reduces overall robustness versus an internal PCIe/NVMe slot.proxmox​

Recommended usage

Using a USB NVMe as the Proxmox system and VM storage disk is reasonable for homelab, testing, and non‑critical workloads where you accept some extra risk and complexity. For production or always‑on services, an internal SSD/NVMe with proper power and cabling is strongly preferred, potentially combined with a separate small device (internal SATA/SD) for the bootloader on hardware that cannot boot directly from NVMe.proxmox+3​

You can set up GPU passthrough in Proxmox, but it will not be automatic just because you’re booting Proxmox from a USB NVMe or using CachyOS. Passthrough requires explicit, per-VM configuration and a properly prepared host with IOMMU support and well‑configured PCI device isolation. Step-by-step plan

  • Verify hardware IOMMU readiness
    • Confirm AMD IOMMU is enabled in BIOS/UEFI (AMD-Vi). Then ensure the kernel sees IOMMU by checking dmesg for DMAR/IOMMU lines after boot. If not present, enable AMD IOMMU in the kernel boot parameters (amd_iommu=on) and update the bootloader, then reboot. [General guidance; adapt to your system]
  • Prepare Proxmox host for PCI passthrough
    • Enable IOMMU in the Proxmox host (same amd_iommu=on parameter for AMD CPUs) and ensure the VFIO modules load: vfio, vfio_pci, vfio_iommu_type1. Update initramfs and reboot. This is foundational for isolating the GPU to a VM.
  • Identify IOMMU groups and the GPU PCI devices
    • Run a PCI scan (lspci -nn) to locate the RTX 4070 and its HDMI/HD audio controller, and note their IDs (for example: 01:00.0 [10de:1e87] VGA controller, 01:00.1 [10de:10f0] Audio). Also check IOMMU groupings (for example, via: for d in /sys/kernel/iommu_groups/*; do echo "IOMMU group:"; ls -1 "$d"; done). If they share an IOMMU group with other devices, you may need ACS override or reordering PCIe slots. This step is critical for stable passthrough.
  • Configure PCI passthrough for the VM
    • In Proxmox, create or edit a Windows or Linux VM and add the NVIDIA GPU as a PCI device (and its associated audio device if present). Use PCI passthrough mode and ensure the guest has appropriate drivers (NVIDIA drivers for Windows or Linux VGPU-compatible stack for Linux). If you encounter boot issues, you may need to add kernel boot parameters to the guest (e.g., except for the host IOMMU, enable vfio-pci binding) and adjust IOMMU remapping settings on the host.
  • Optimize and test
    • Boot the VM and verify the GPU initializes inside the guest (device manager in Windows or lspci inside Linux). Install the correct NVIDIA driver inside the VM. Run a basic GPU test to confirm full GPU access. If the host becomes unresponsive or IOMMU groups are too coarse, revisit ACS overrides or PCI slot configuration.
  • Stabilize and maintain
    • If stability is an issue, consider aligning the VM with a hot-plug PCI device policy, keeping the host’s PCIe topology favorable (prefer a dedicated GPU PCIe slot, avoid sharing with critical host IO devices), and ensuring robust power and cooling for both host and VM workloads.

Notes and caveats

  • GPU passthrough success depends on clean IOMMU groups; if the RTX 4070 shares a group with other devices, passthrough may be unstable or impossible without enabling more aggressive ACS overrides. This remains the most common blocker on many systems.
  • Running GPU passthrough on a USB NVMe boot drive does not inherently enable or disable passthrough; it’s orthogonal to the task. The key is ensuring the host’s IOMMU stack and PCI devices are properly isolated and assigned to the VM.
  • For AMD, you’ll likely rely on amd_iommu=on and potentially pcie_acs_override=downstream,multifunction to improve grouping, but exact parameters depend on the motherboard and firmware. Start with the standard AMD IOMMU guidance and tailor via testing.
