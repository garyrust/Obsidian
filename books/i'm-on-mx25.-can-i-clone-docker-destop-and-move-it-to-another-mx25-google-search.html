Yes, you can clone your Docker Desktop data and move it from one

MX 25 system to another. However, you cannot simply copy the application itself; you must move the images, containers, and volumes.

Option 1: Move the Entire Virtual Disk (Easiest)

On Linux, Docker Desktop stores its data (images and volumes) in a virtual disk file.

  1. Locate the disk image: Open Docker Desktop, go to Settings > Advanced, and find the path under Disk image location.
  2. Stop Docker Desktop: Completely quit the application on both machines.
  3. Transfer the file: Copy the disk image file (often a .raw or .vhdx file) to the same location on the new MX 25 machine.
  4. Restart: Launch Docker Desktop on the new machine. It should automatically recognize your previous environment.

Option 2: Export/Import Specific Parts (Safest)

If you only need certain projects, use individual commands to ensure data integrity:

  • Images: Use docker save -o my_image.tar <image_name> to create a backup file. On the new machine, use docker load -i my_image.tar.
  • Containers: Commit a running container to an image first: docker commit <container_id> snapshot_name. Then export that snapshot using the image method above.
  • Volumes: Use the Volumes Backup and Share extension in Docker Desktop to export volumes as .tar.gz files and import them on the new system.

Important Considerations

  • Architecture: If you are moving from an x86 machine to an ARM machine (or vice-versa), your existing images may not run.
  • Bind Mounts: If your containers use local folders on your MX 25 host (e.g., -v /home/user/data:/data), you must manually copy those host folders to the exact same path on the new machine.
  • MX Snapshots: Since you are on MX Linux, you could also use the MX Snapshot tool to clone your entire OS to a live USB, which will include your Docker setup if you include the home/root directories.

Yes, you can clone your entire Docker Desktop workspace from one MX 25 system to another by moving its dedicated

virtual disk image. This is the most efficient way to migrate all images, containers, and volumes at once.

Step 1: Locate the Workspace File

Docker Desktop on Linux does not store data in the standard /var/lib/docker path used by native Docker Engine. Instead, it uses a virtual disk file:

  1. Open Docker Desktop on your original MX 25 machine.
  2. Navigate to Settings > Resources > Advanced.
  3. Note the path under Disk image location. It is usually a file named Docker.raw located within your home directory (e.g., ~/.docker/desktop/vms/default/).

Step 2: Transfer the Data

To ensure a clean migration without data corruption:

  1. Quit Docker Desktop on both the old and new MX 25 machines.
  2. Copy the file: Transfer the Docker.raw (or similarly named) file to the new machine using an external drive or network transfer.
    • Tip: These files can be very large. Compressing them first (e.g., zip Docker.raw.zip Docker.raw) can save significant transfer time.
  3. Place the file: On the new MX 25 machine, move the file to the exact same directory path it occupied on the old machine.

Step 3: Finalize on the New Machine

  1. Launch Docker Desktop on the new system.
  2. If the workspace doesn't appear immediately, go back to Settings > Resources > Advanced, click Browse, and manually select the transferred disk image file.
  3. Click Apply & Restart.

Alternative: MX Snapshot (Full System Clone)

If you want to move your entire MX 25 environment—including your Docker Desktop installation, configurations, and all other files—use the built-in MX Snapshot tool.

  • Open MX Snapshot from your menu.
  • Select Preserve accounts (for personal backup) to include your home directory and Docker data.
  • This creates a bootable ISO file that you can use to install an identical copy of your current system onto the new machine.

Note on Hardware: If you are moving from an Intel/AMD (x86_64) machine to an ARM machine (like a Raspberry Pi or certain laptops), your existing images will not work because of the different CPU architecture.

To move your entire workspace including external

bind mounts (folders on your MX 25 host linked to containers), you must ensure those exact folder paths and their data are mirrored on the new machine.

1. Identify Your Bind Mounts

2. Move the Data (Mirroring Paths)

For bind mounts to work automatically after you move your Docker.raw file, the folders must exist at the exact same absolute path on the new machine.

  • Use Rsync (Recommended): To preserve permissions and ownership (critical for databases or web servers), use rsync:
    rsync -avz /path/to/original/folder/ user@new-mx25:/path/to/original/folder/
  • Manual Copy: If using a USB drive, copy the folders and ensure the destination path on the new MX 25 matches the old one exactly (e.g., if it was in /home/user/my-project, it must stay there).

3. Verification & Fixes

Once you've moved both the Docker disk image (from the previous step) and your host folders:

  1. Launch Docker Desktop on the new machine.
  2. Check Permissions: If a container fails to start or can't write to a folder, the User ID (UID) on your new machine might be different. You may need to update ownership:
    sudo chown -R $USER:$USER /path/to/your/folder
  3. Docker Compose: If you use Docker Compose, simply run docker-compose up -d from your project folder. It will automatically reconnect to the newly moved host folders.

Summary Checklist for a "Perfect" Clone

  • Stop Docker Desktop on both machines.
  • Copy Docker.raw to the same hidden directory on the new machine.
  • Copy all bind-mount host folders to their identical paths.
  • Start Docker Desktop and verify containers are running.
